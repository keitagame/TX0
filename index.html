§ 1

概要と設計思想

TXTPC-0（テキストコンピューター・ゼロ）は、命令数を極限まで削減しながら、現代のパーソナルコンピューターに匹敵する機能を実現することを目標として設計された架空のテキストベース計算機械です。

全命令数はわずか 8 命令のみ。しかしその 8 命令は、テキスト処理・ファイルシステム・ネットワーク通信・タイマー・テキストUI・プロセス管理に至るまで、あらゆる操作を表現できるよう設計されています。

秘訣は「すべてをテキストストリームとして統一する」という設計哲学の徹底です。ファイルも、ネットワーク接続も、デバイスも、プロセス間通信も、すべて「名前付きストリーム」として同一の命令で扱います。命令の少なさは制約ではなく、設計の純粋さの証明です。

§ 3

命令セット（ISA）

TXTPC-0 の命令セットは下記の 8 命令 のみで構成されます。オペコードは 3 ビット（0〜7）で表現されます。すべての命令はテキストストリームの読み書き・制御を基本とし、それ以外の一切の概念を持ちません。

命令名 オペコード 構文 動作説明
OPEN 000 OPEN <名前> Rx 名前付きストリームを開き、ハンドルをレジスタ Rx に格納する。ファイル・デバイス・ネットワーク・プロセスすべて同一形式。
READ 001 READ Rx Ry Rx のストリームから次の「トークン」（空白 / 改行区切り）を読み込み、テキストとしてレジスタ Ry に格納する。
WRITE 010 WRITE Rx Ry レジスタ Ry のテキストを Rx のストリームへ書き込む。画面出力も、ネットワーク送信も、ファイル保存もこれ一つ。
COPY 011 COPY Rx Ry レジスタ Rx の内容をレジスタ Ry にコピーする。即値テキストをレジスタに格納する際にも使用。
MATCH 100 MATCH Rx Ry Rz Rx のテキストを Ry のパターン（正規表現）でマッチングし、結果（一致テキストまたは空文字）を Rz に格納する。条件分岐の唯一の基盤。
JOIN 101 JOIN Rx Ry Rz Rx と Ry のテキストを連結し Rz に格納する。数値演算もテキスト変換後にこれで式文字列を組み立てる。
JUMP 110 JUMP Rx <ラベル> Rx が空文字でない場合、指定ラベルへ制御を移す。MATCH と組み合わせることで if / while / for をすべて表現できる。
CLOSE 111 CLOSE Rx Rx のストリームを閉じる。プロセス終了もこれで行う（標準出力ストリームをクローズ）。
IMPORTANT 算術演算命令は存在しません。数値はすべてテキストとして表現し、特殊ストリーム sys:calc に式文字列を WRITE → READ することで計算結果を取得します。これにより整数・浮動小数点・任意精度演算が統一的に扱えます。
§ 4

アーキテクチャ

TXTPC-0 のアーキテクチャは「ストリームバス」を中心に構成されます。CPU・メモリ・すべての周辺機能は、ストリームバスを介してテキストを交換するだけで動作します。

レジスタ仕様

16 本の汎用レジスタ（R0〜R15）は、それぞれ任意長の UTF-8 テキスト文字列を保持します。固定ビット長ではなく「テキストバッファ」として機能するため、1 文字でも 1 MB の文書でも同じように扱えます。スタックポインタ（SP）とプログラムカウンタ（PC）も内部的にはストリーム位置を示すテキストアドレスです。

ストリーム名前空間

あらゆるリソースを以下の名前空間プレフィックスで統一的に参照します。

プレフィックス 対象リソース 記述例
fs: ファイルシステム fs:/home/user/doc.txt
net: ネットワーク通信（TCP/UDP） net:tcp://192.168.1.1:8080
http: HTTP リクエスト http:GET:https://example.com
sys: システムサービス sys:calc, sys:timer, sys:proc
dev: デバイス dev:keyboard, dev:display
pipe: プロセス間通信 pipe:my-channel
clip: クリップボード clip:system
enc: 暗号化レイヤー（重ね掛け可） enc:aes256:fs:/secret.txt
§ 5

メモリマップ

仮想アドレス空間（48 bit）は以下のように区分されます。物理メモリの割り当ては実装依存ですが、論理レイアウトはすべての実装で共通です。

アドレス先頭 領域名 サイズ 用途
0x0000_0000_0000 ゼロページ 4 KB NULL 参照検出・アクセス禁止領域
0x0000_0000_1000 ブートROM 64 KB 起動スクリプト・初期ストリーム名前解決テーブル
0x0000_0001_0000 システム領域 16 MB カーネルストリームバス・デバイスドライバ
0x0000_0100_0000 テキストキャッシュ 256 MB 頻繁にアクセスされるストリームのバッファリング
0x0000_1100_0000 ユーザー空間 〜 128 TB プログラム・スタック・ヒープ（テキスト形式）
0x7FFF_FFFF_F000 ストリームハンドル表 256 MB 最大 65536 本のオープンストリーム管理テーブル
§ 6

入出力・通信機能

TXTPC-0 の入出力はすべてストリーム経由です。命令を追加することなく、以下の機能を完全に実現します。

テキスト端末 標準
dev:tty ストリームへの READ / WRITE で、キーボード入力・画面出力を行う。VT100 エスケープシーケンスも文字列として送出するだけでカーソル制御・色付けが可能。
TCP / UDP 通信 標準
net:tcp://host:port を OPEN するだけでソケット接続確立。WRITE でデータ送信、READ でデータ受信。TLS も net:tls:// で透過対応。
HTTP クライアント 標準
http:GET:https://url を OPEN → READ でレスポンスボディ取得。POST は WRITE でボディ送信後に READ。REST API 呼び出しが 3 命令で完結する。
WebSocket 標準
net:ws://host/path をオープンすることで双方向リアルタイム通信。READ / WRITE を繰り返すだけでメッセージ交換できる。
メール送受信 標準
net:smtp:// ストリームに宛先・件名・本文をテキストとして WRITE するだけ。受信は net:imap:// ストリームから READ。MIMEエンコードはシステムが自動処理。
DNS 問い合わせ 標準
sys:dns に「resolve example.com」と WRITE → READ で IP アドレス取得。逆引き・MX レコードも同様のテキスト形式で照会できる。
Bluetooth / シリアル オプション
dev:bt:XX:XX:XX で Bluetooth デバイスとテキスト通信。シリアル通信は dev:serial:/dev/ttyS0 で同一インターフェース。
プロセス間通信 標準
名前付きパイプ pipe:名前 を OPEN するだけで複数プロセスがリアルタイムにテキストを共有。セマフォも「1」/「0」の READ / WRITE で実現。
§ 7

記憶媒体

TXTPC-0 は単一の fs: 抽象を通じて、あらゆる物理記憶媒体を透過的に扱います。マウント先のパスを変えるだけで、同じコードが異なるストレージで動作します。

揮発性 RAM ディスク
最高速。fs:/ram/ にマウント。作業中のテキストバッファ・一時ファイルに使用。電源断で消去される。
フラッシュ / SSD
fs:/ssd/ にマウント。主記憶。書き込みは非同期キャッシュに蓄積後、バックグラウンドでフラッシュされる。
ネットワークストレージ
fs:/net/share/ にマウント。実体は NFS / SMB / SFTP 等。アクセスはローカルと完全に同一の命令で行える。
クラウドオブジェクトストレージ
S3 互換ストレージを fs:/cloud/bucket/ にマウント。大規模テキストデータの保存・配信に対応。
暗号化ボリューム
enc:aes256:fs:/secret/ と名前空間を重ねるだけで透過暗号化。鍵管理は sys:keyring 経由。
バージョン管理 FS
fs:/vcs/repo/ にマウントすると、すべての WRITE が自動的にバージョン記録される。差分も READ で取得可能。
§ 8

プログラム例

例 1 ─ 「Hello, World!」を画面に出力

OPEN  dev:tty        R0    ; 端末ストリームを開く
COPY  "Hello, World!\n" R1  ; テキストをレジスタにセット
WRITE R0 R1               ; 端末へ書き込む
CLOSE R0                  ; ストリームを閉じる

例 2 ─ HTTP GET してファイルに保存

OPEN  http:GET:https://example.com/data.txt  R0  ; HTTP ストリームを開く
OPEN  fs:/home/user/data.txt                 R1  ; 保存先を開く
:loop
  READ  R0 R2            ; レスポンスを 1 トークン読む
  JUMP  R2 :write        ; 内容があれば書き込みへ
  JUMP  R0 :done         ; なければ終了
:write
  WRITE R1 R2            ; ファイルへ書き込む
  JUMP  "1" :loop        ; 常にループ継続
:done
CLOSE R0
CLOSE R1

例 3 ─ 数値の加算（sys:calc を使用）

OPEN  sys:calc  R0    ; 計算サービスストリームを開く
COPY  "37"      R1
COPY  "85"      R2
JOIN  R1 "+"    R3    ; "37+" を作る
JOIN  R3 R2     R4    ; "37+85" を作る
WRITE R0 R4           ; 式を送信
READ  R0 R5           ; 結果 "122" を受信
CLOSE R0

例 4 ─ TCP サーバー（接続を受け付けて応答）

OPEN  net:tcp:listen:0.0.0.0:8080  R0  ; 待受ストリームを開く
:accept
  READ  R0 R1        ; 接続を受け入れる（ハンドルが R1 に）
  JUMP  R1 :handle
  JUMP  "1" :accept
:handle
  READ  R1 R2        ; リクエストを読む
  COPY  "OK\n" R3
  WRITE R1 R3        ; レスポンスを返す
  CLOSE R1           ; 接続を閉じる
  JUMP  "1" :accept  ; 次の接続へ
§ 9

設計哲学ノート

「なぜ 8 命令で十分なのか」という問いへの答えは、コンピューターが本質的に行う操作は「どこかから読んで、どこかへ書く」という一つのことだけである、という洞察にあります。

命令セットが小さいことは、実装の正確さを高めます。8 命令のデコーダは検証が容易であり、セキュリティ脆弱性の攻撃面積が最小化されます。プロセッサのゲート数が少ないため、超低消費電力デバイスへの搭載も容易です。

一方で「多機能」を実現するのは命令ではなくストリーム名前空間の豊かさです。新しい機能（例：量子乱数生成）を追加したい場合、命令セットに手を加える必要はなく、sys:qrng という名前のストリームを新設するだけです。命令セットは完全に安定したまま、システム機能は無制限に拡張できます。

DESIGN PRINCIPLE 命令の少なさと機能の豊かさはトレードオフではない。抽象化のレイヤーを適切に分けることで両立できる。TXTPC-0 はその証明である。
